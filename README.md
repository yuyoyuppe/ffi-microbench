# ffi-microbench — csbindgen vs uniffi-bindgen-cs

Microbenchmarks for the C# ↔ Rust FFI boundary, comparing three surfaces exposed by
**one** Rust cdylib (identical compiler settings: `--release`, LTO, `codegen-units = 1`):

1. **csbindgen** — hand-rolled `extern "C"` exports consumed through
   [csbindgen](https://github.com/Cysharp/csbindgen)-generated P/Invoke, replicating the
   conventions of a production Windows app's `native-dll` (UTF-16 ptr+len strings in, owned UTF-8
   `CString` out, JSON for compound payloads, bare Cdecl fn-pointer callbacks, bool results).
   This is the incumbent/baseline.
2. **uniffi (stock)** — `#[uniffi::export]` surface consumed through
   [uniffi-bindgen-cs](https://github.com/NordSecurity/uniffi-bindgen-cs) with default config.
3. **uniffi (span fast path)** — same bindings generated with the
   [`high_performance_strings`](https://github.com/yuyoyuppe/uniffi-bindgen-cs/tree/span_feature)
   fork option: string arguments cross as borrowed UTF-8 `ReadOnlySpan<byte>` via `*_raw`
   exports instead of RustBuffer copies, and standard string overloads transparently delegate
   to the span variants.

The case matrix follows `uniffi_migration_benchmarks.md` (the Windows FFI migration plan);
results serve as evidence for upstream PRs against uniffi-bindgen-cs.

## Quick start

```powershell
# everything: build, codegen, smoke verification, full benchmark run
.\run_benchmarks.ps1

# fast iteration / subsets
.\run_benchmarks.ps1 -Quick -Filter *Case04*
.\run_benchmarks.ps1 -SmokeOnly

# stability/leak stress (multi-threaded, non-ASCII payloads, object churn,
# concurrent callbacks, exception paths; fails on memory drift)
.\run_benchmarks.ps1 -SmokeOnly -Stress -StressSeconds 30
```

Prereqs: Rust (1.88+), .NET SDK 8+, pwsh 7, and a checkout of the uniffi-bindgen-cs fork
(default: sibling directory `..\uniffi-bindgen-cs`, override with `-BindgenRepo`).

Reports land in `results\results\*-report-github.md`; toolchain + commit provenance in
`results\environment.txt`.

## Layout

```
run_benchmarks.ps1          single entry point (build → codegen → smoke → bench)
rust/
  src/api_csb.rs            csbindgen surface (incumbent conventions, no catch_unwind)
  src/api_uniffi.rs         uniffi 0.31 proc-macro surface
  src/api_raw.rs            hand-written *_raw exports for the span fast path
  src/types.rs              shared payload types + leaked static test data
  uniffi-stock.toml         stock codegen config    -> namespace uniffi.benchffi
  uniffi-span.toml          span-fast-path config   -> namespace uniffi.benchffi_span
dotnet/FfiBench/
  Benchmarks/Case01..14*.cs one class per matrix case, BenchmarkDotNet
  Smoke.cs                  cross-surface correctness gate (runs before any timing)
  Stress.cs                 multi-threaded stability/leak harness (-Stress)
  Generated/                committed codegen output (regenerated by the script):
    NativeMethods.g.cs      csbindgen
    stock/benchffi.cs       uniffi-bindgen-cs, default config
    span/benchffi.cs        uniffi-bindgen-cs, high_performance_strings = true
```

## Methodology / parity notes

- **One cdylib.** All three surfaces compile into `benchffi.dll`; no compiler-settings skew.
- **Checksummed work.** Every `take_*` returns a value derived from all elements; `give_*`
  clones lazily-built leaked statics, so timings measure the FFI-mandated copy, not payload
  generation. Payloads are ASCII so UTF-16 length == UTF-8 length across surfaces.
- **Smoke gate.** `--smoke`/`--smoke-span` run every operation once on every surface and
  cross-check results; the runner refuses to benchmark if any check fails.
- **NET8 paths active.** The bench targets net8.0, so the bindings'
  `#if NET8_0_OR_GREATER` fast paths (source-generated `[LibraryImport]`, span-based
  `BigEndianStream` reads) are compiled in — enforced by an `#error` guard in `Program.cs`.
- **Flavor isolation.** Stock and span bindings of the same interface register the same
  Rust-side callback vtable slots; the two smoke phases therefore run in separate processes,
  and every BenchmarkDotNet case runs in its own child process (default toolchain — do not
  switch callback benchmarks to an in-process toolchain).
- **Allocations are first-class.** `MemoryDiagnoser` is always on; per-invocation managed
  allocations matter as much as ns/op for per-keystroke/log-line callbacks.
- The doc asks for median + p99; BenchmarkDotNet ships P90/P95 percentile columns, and the
  full distribution is in `results\results\*-report-full.json`.

## Regenerating bindings

`run_benchmarks.ps1` regenerates `Generated/` on every run (skip with `-SkipCodegen`).
The committed copies exist so the suite is reviewable/diffable without running codegen.
